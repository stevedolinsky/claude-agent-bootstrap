# Brainstorm: Workers Should Use Target Repo Context

**Date:** 2026-03-25
**Status:** Draft

## What We're Building

Align the worker execution model with setup.sh's original intent: workers should run in the context of the target repo (e.g., permitradar), picking up its CLAUDE.md, `.claude/settings.json`, and local files. Currently workers always run from the bootstrap directory and miss the target repo's agent instructions.

## The Problem

`setup.sh` generates CLAUDE.md with agent instructions (verify chain, comment signatures, labels, MCP-first rule) in the target repo. But the centralized receiver runs from `~/claude-agent-bootstrap/`, and workers inherit that CWD. Workers never see the target repo's CLAUDE.md.

This caused a production regression on 2026-03-24: the bootstrap repo's local CLAUDE.md (accidentally created by running setup.sh there) was wiped by a `git reset origin/main`, and workers stopped posting comments entirely.

## Why This Approach

Following Claude Code best practices:
- **CLAUDE.md is the behavioral source of truth** — it tells Claude how to work in a project (generated by setup.sh in the target repo)
- **The prompt provides the specific task** — what to do right now (from `_build_prompt()`)
- **MCP tools provide capabilities** — configured via .claude/settings.json (in the target repo)

This means:
- Workers should run with `--directory <target_repo>` so Claude loads the right CLAUDE.md and settings
- `_build_prompt()` only needs to provide task-specific context (comment body, author, PR number) — not behavioral instructions
- The existing templates (`orchestrator.md`, `pr-responder.md`) are unnecessary — CLAUDE.md handles behavior, the prompt handles the task

## Key Decisions

1. **`--directory <target_repo>` on worker command** — Claude loads the target repo's CLAUDE.md and `.claude/settings.json` automatically. This is how Claude Code is designed to work.

2. **`_build_prompt()` provides only task context** — "Respond to PR comment on #425 from stevedolinsky: '...'" — not behavioral instructions. Simple string construction, no template rendering needed.

3. **Templates are unnecessary** — CLAUDE.md carries behavioral instructions, prompt carries task context. Templates can be kept as documentation/reference but are not rendered at runtime.

4. **Repo path registered in TOML by setup.sh** — When you run `setup.sh` in a target repo, it appends the repo's path to `~/.claude/agent-receiver.toml`:
   ```toml
   [repos."stevedolinsky/permitradar"]
   path = "/home/sdolinsky/permitradar"
   ```
   Then restarts the receiver. Workers are separate process groups — in-flight workers survive the restart.

5. **Graceful fallback for unregistered repos** — If a webhook arrives for a repo not in the TOML, the worker runs without `--directory` (degraded — uses bootstrap CLAUDE.md). Functional but missing target-specific instructions.

6. **Two new QueueItem fields** — `comment_body: str = ""` and `comment_author: str = ""`. Set at enqueue time from the webhook payload. These are the minimum data `_build_prompt()` needs beyond what QueueItem already has.

7. **No new dependencies** — No template library needed. `_build_prompt()` uses f-strings. Config reads TOML (already a dependency).

## Resolved Questions

- **Where does verify_chain come from?** → From the target repo's CLAUDE.md (loaded via --directory). Not needed in the prompt.
- **Where does pr_branch come from?** → The worker reads it via MCP tools (`mcp__github__pull_request_read`). Not needed in the prompt — Claude can fetch it.
- **What about multi-repo?** → setup.sh registers each repo in TOML and restarts the receiver. Per-repo queues already handle isolation. In-flight workers survive restarts.
- **What if the target repo path changes?** → Re-run setup.sh from the new location. It overwrites the TOML entry.
